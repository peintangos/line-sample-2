### スライド 2: なぜ LINE を入口に選ぶのか

AIエージェントの入口は何でもいい。Webアプリ、Slack、専用アプリ。
その中で LINE を選ぶ理由は1つ。

**ユーザーが毎日使っているアプリに、AIエージェントの入口を置ける。**
新しいアプリのインストールも、アカウント登録も要らない。
日本で約9800万人の生活導線に直接届く。

一方で、Messaging API には制約がある:
- Streaming（1文字ずつ流れる表示）ができない
- 送ったメッセージを後から更新できない
- チャット内に自由なUIを埋め込めない

**今日は、この制約の中で最先端のエージェント体験にどこまで寄せられるかを話す。**

話す内容:
- 理由はシンプル。9800万人がすでにいる場所だから
- でも Messaging API はAIエージェント向けに設計されていない。制約がある
- 今日はこの制約の中でどう寄せるかの話

表現:
- 理由は一言で大きく
- 制約は箇条書きで正直に

### スライド 3: 最先端のAIエージェントで「入口」に起きていること

ユーザーがAIエージェントと接する場面で、いま当たり前になりつつある体験:

| 体験 | 例 | ユーザーにとっての価値 |
|------|-----|----------------------|
| Streaming | ChatGPT / Claude のトークン逐次表示 | 待たされている感じがしない |
| 動的UI | AIがカード・グラフ・フォームを返す（MCP Apps） | テキストだけで終わらない |
| 実行前の承認 | 「この操作を実行していい？」 | 勝手に動かれない安心感 |
| 画像理解 | 写真を渡すとAIが中身を理解する | 言葉で説明しなくていい |

**動的UI の最前線: MCP Apps**

- MCP（Model Context Protocol）の初の公式拡張仕様（2026年1月リリース）
- MCP サーバーが **インタラクティブな HTML UI を AI チャット内に直接表示できる**
- Anthropic と OpenAI が共同策定したオープン標準
- Claude / ChatGPT / VS Code 等が対応済み
- Amplitude, Figma, Slack, Salesforce 等10社がローンチパートナー

つまり: AI エージェントが「テキストで返事する」から **「UIごと返す」** 時代に入った。
ダッシュボード、フォーム、ビューワーが会話の中に埋め込まれる。

話す内容:
- プロトコル名（MCP, AG-UI, A2UI）は覚えなくていい
- でも MCP Apps は知っておくと良い。AI の返事がテキストからUIに変わりつつある
- 大事なのはユーザーが「入口」で何を期待するか
- この4つに対して、Messaging API でどう表現するかを見ていく

表現:
- 4つの体験をカード形式で並べる
- MCP Apps のスクリーンショット（Claude の会話内にダッシュボードが表示されている例）を右側に

### スライド 4: Messaging API でできること・できないこと

| 体験 | Messaging API での現実 | なぜできないか |  |
|------|------------------------|--------------|--|
| Streaming | メッセージは完成形で1通ずつ届く | レスポンスをチャンク送信する API がない。SSE / WebSocket 非対応。AI がトークンを生成するたびに画面に追記する体験は不可能 | 🔴 |
| 動的UI | 送ったメッセージ・Flex の内容は二度と変わらない | 送信済みメッセージを編集・更新する API がない（Telegram の `editMessageText` に相当するものがない）。ボタンの状態変更、カード内テキストの書き換え、すべて不可 | 🔴 |
| 実行前の承認 | Quick Reply / Postback で選択肢を出せる | — | 🟡 |
| 画像理解 | 画像送受信は標準対応。Vision AI に渡せる | — | 🟢 |

見落としがちな制約:
- Reply Token は数十秒で失効。重い処理は Push Message（非同期）にする
- 無料プランは月200通の Push Message 制限
- 会話履歴APIがない。文脈維持はサーバー側で自前構築

話す内容:
- 🔴 の2つは API が存在しないのでどうしようもない。代替で寄せるしかない
- Streaming ができないのは「チャンク送信の仕組みがない」から
- 動的UI ができないのは「送信済みメッセージを書き換える API がない」から
- この2つの壁をどう乗り越えるかがこの後の本題

表現:
- 🔴🟡🟢 で色分け
- 🔴 の「なぜ」を太字 or 吹き出しで補足して解像度を上げる

### スライド 5: 寄せ方の全体像

**できないことは割り切る。できることを組み合わせて、入口としての体験を作る。**

| やりたいこと | 使う API / 機能 |
|-------------|-----------------|
| 会話応答 | Reply Message / Push Message |
| 構造化された情報表示 | Flex Message（AI が JSON を生成） |
| ユーザーの承認・選択 | Quick Reply + Postback |
| 「考え中」の表示 | Loading Animation |
| 役割の見せ分け | sender（アイコン・名前切替） |
| 画像分析 | 画像受信 → getMessageContent → Vision AI |

話す内容:
- 全部トーク画面内で動く
- LIFF（LINE 内 WebView）を使えば Streaming もできるが、チャットから離れるので今日は扱わない
- トーク画面の中だけでどこまでいけるかにフォーカスする

表現:
- テーブルを「やりたいこと → 使う API」のマッピングとして見せる
- 「トーク画面内で完結」を強調

### スライド 6: 寄せ方① — Streaming → Loading + 非同期 Push + Flex

1文字ずつ流れる体験は作れない。代わりに:

1. ユーザーのメッセージ受信
2. **Loading Animation** を表示（「...」が動く。最大60秒）
3. サーバー側で AI に問い合わせ
4. 結果を **Push Message** で送信。長い回答は **Flex Message** で構造化

「ChatGPT 風の再現」ではなく「LINE 的な AI 応答」を作る:
- 考え中であることは伝わる
- 結果は構造化されて見やすく届く
- ユーザーは「待って→答えが来た」で十分成立する

話す内容:
- Streaming は割り切る。でも Loading + Flex で実用上は成立する
- Loading Animation は1対1チャットのみ。グループでは動作しない

表現:
- 3ステップのフロー図: 受信 → Loading → Flex で結果表示

### スライド 7: 寄せ方② — 動的UI → Flex Message を AI が毎回生成する

送った後に UI が変化する体験は作れない。代わりに:

- AI の **Structured Output** で Flex Message の JSON を毎回動的に生成する
- リクエストごとに内容が変わる = **「内容が動的な」カード**
- 送信後に変化はしないが、その時点の最適な表示が毎回届く

活用例:
- 検索結果 → カルーセル形式の Flex
- 予約確認 → 日時・場所・金額をまとめたカード
- 分析結果 → セクション分けされた要約カード

話す内容:
- Flex Message は CSS Flexbox ベースの JSON で、表現力はかなりある
- AI に JSON を直接生成させるのは有効なパターン
- 本物の Generative UI との違い: 送った後に状態が変わるかどうか。でも多くのケースでは「その時点の情報が届く」で足りる

表現:
- AI → Structured Output → Flex Message JSON → LINE カード表示の流れ
- Flex Message の実例スクリーンショット

### スライド 8: 寄せ方③ — 実行前承認 → Quick Reply + Postback

エージェントがツールを実行する前に、ユーザーの承認を挟む:

> Bot: 「明日14時にミーティングを登録します」（Flex Message で詳細）
> Quick Reply: [承認] [時間変更] [キャンセル]
> ユーザー: [承認] をタップ
> Bot: 「登録しました ✅」

| 要素 | 使う API |
|------|----------|
| 内容の提示 | Flex Message |
| 選択肢 | Quick Reply |
| ユーザーの選択 | Postback |
| 日時入力 | Datetime Picker アクション |
| 状態管理 | サーバー側にセッション ID で保持 |

話す内容:
- 最先端では `needsApproval` フラグ一つで承認UIが自動生成される
- LINE では自分で組み立てる必要があるが、フローとしては同等のことができる
- Quick Reply はタップ1つで完了するシンプルさが逆に強み

表現:
- LINE のトーク画面風モックで承認フローを見せる

### スライド 9: 寄せ方④ — 画像理解 + sender 切替

**画像理解は LINE の得意分野:**
- 画像・動画・音声・ファイルの送受信が標準装備
- `getMessageContent` → Vision AI → Flex Message で結果を返す
- 「写真を撮って送るだけ」— インストールも追加UIも不要
- 例: 料理写真→カロリー推定、名刺→連絡先カード、レシート→家計簿登録

**sender 切替で役割を可視化:**
- `sender` プロパティでメッセージごとにアイコンと名前を変えられる
- 「検索」「分析」「確認」など、処理の役割をユーザーに見せられる
- ただし A2A（エージェント間プロトコル）ではない。内部は1つの Bot

話す内容:
- 画像理解は最先端と遜色ない。むしろ「送るだけ」の導入障壁の低さは LINE ならでは
- sender は A2A の再現ではなく、処理の可視化として使う

表現:
- 左: 料理写真 → カロリーカードのフロー
- 右: sender 切替で異なるアイコンの吹き出しが並ぶ画面

### スライド 10: 補足 — LIFF と Telegram という選択肢

**LINE 内でもっと自由にやるなら: LIFF**

- LIFF（LINE Front-end Framework）はトーク内で Web アプリを開く仕組み
- Web アプリなので **Streaming も動的 UI も自由に実装できる**
- ただし WebView なので、トーク画面のネイティブ体験からは離れる
- 「チャットの中にいる感」は薄れるが、表現の天井はなくなる

**別のプラットフォームなら: Telegram Bot API**

Telegram は AI エージェントの入口として非常に優秀。

| 比較軸 | LINE Messaging API | Telegram Bot API |
|--------|-------------------|------------------|
| 通信方式 | Webhook のみ | **Long Polling (`getUpdates`) + Webhook** の2方式（排他的） |
| メッセージ編集 | 不可 | **送信済みメッセージを後から編集可能**（`editMessageText` 等） |
| 疑似 Streaming | 不可 | **editMessageText の繰り返しで実現可能** |
| ボタン更新 | Quick Reply はタップで消える | **インラインキーボードは送信後も残り、更新可能** |
| Mini Apps | LIFF（WebView） | **Telegram Mini Apps**（フルスクリーン、デバイスセンサー、決済、生体認証） |
| 開発体験 | 常に公開 Webhook サーバーが必要 | **Long Polling ならローカルで即開発可能** |

> Telegram は `editMessageText` でメッセージを後から書き換えられる。
> これにより AI の逐次出力を1つの吹き出し内で更新でき、Streaming 風の体験が作れる。
> LINE の制約の多くは、Telegram では制約にならない。

話す内容:
- 自分は HermesAgent という AI エージェントの対話入口として Telegram を使っている
- LINE を選ぶ理由は9800万人のリーチ。Telegram を選ぶ理由は表現力と開発体験
- どちらが優れているではなく、目的で選ぶ

表現:
- LINE vs Telegram の比較テーブルを大きく
- 「目的で選ぶ」を強調

### スライド 11: まとめ

**Messaging API で AI エージェントの入口をどこまで作れるか。**

| 最先端の体験 | Messaging API での表現 | どこまでいけるか |
|-------------|----------------------|-----------------|
| Streaming | Loading + 非同期 Push + Flex | 体験は異なるが実用上は成立する |
| 動的UI | Flex Message を AI が毎回生成 | 内容は動的。送信後の変化は不可 |
| 実行前承認 | Quick Reply + Postback | フローとしてはほぼ同等 |
| 画像理解 | 画像受信 + Vision AI | 最先端と遜色なし |
| 役割の可視化 | sender 切替 | 見せ方として有効。A2A ではない |

> LINE はAIエージェントの実行環境ではなく、ユーザーとの接点。
> Messaging API には制約があるが、既存機能の組み合わせで入口としての体験は作れる。
> 「ChatGPT の再現」ではなく「LINE の入口」として設計する。

表現:
- まとめテーブルを大きく表示
- QRコード: 記事 + ショーケース Bot + GitHub
