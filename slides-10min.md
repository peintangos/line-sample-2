## Flex / Quick Reply / sender で最先端にどこまでいけるか

---

> LINE はAIエージェントそのものにはなれない。
> でも「AIエージェントへの入口」としては強い。

### スライド 2: 今日伝えたいこと

LINE は AI エージェントの導入先として入りやすい。
でも最先端の体験をそのまま載せようとすると不便なところがある。
**既存APIの組み合わせで、「入口」としての体験は寄せられる。今日はそのコツを話す。**

「入口」とは:
- AIエージェントの自律的な計画・実行・自己修正の全部をLINEでやる話ではない
- ユーザーがAIエージェントと接する最初の接点 — 指示を出す、結果を受け取る、承認する — をLINEで作る話

| ポイント | 内容 |
|----------|------|
| 導入のしやすさ | 約9800万人がインストール済み。認証不要。通知が届く。新しいアプリを入れさせなくていい |
| 不便なところ | Streaming / 動的UI / エージェント間連携など、最先端では当たり前の体験が素直に作れない |
| でも寄せられる | Flex Message / Quick Reply / Postback / sender 切替などの既存APIで近い体験を組み立てられる |

話す内容:
- これは LINE だけの問題ではない。WhatsApp は汎用AIチャットBotを禁止、Telegram は日本でリーチが弱い
- メッセージングプラットフォーム全般がAIエージェント向けに設計されていない
- その中で LINE を選ぶなら、制約を知った上で寄せる方法を身につけるのが大事

表現:
- 3行のポイントをカード風に縦に積む
- 「寄せられる」を一番大きく

### スライド 3: いまのAIエージェント体験

最先端のAIエージェントで、ユーザーが「入口」で受けている体験を整理する。

| 体験 | 例 | ユーザーにとっての価値 |
|------|-----|----------------------|
| Streaming | ChatGPT / Claude のトークン逐次表示 | 待たされている感じがしない |
| 動的UI生成 | AIがフォーム・グラフ・カードを返す | テキストだけで終わらない |
| 実行前の承認 | 「この操作を実行していい？」 | 勝手に動かれない安心感 |
| ツール接続 | MCP でカレンダー・DB・外部APIに繋がる | AIが実際にタスクをこなす |
| 画像理解 | 写真を渡すとAIが中身を理解する | 言葉で説明しなくていい |

話す内容:
- プロトコル名（MCP, AG-UI, A2UI 等）は知らなくてOK
- 大事なのは「ユーザーがどんな体験を期待しているか」
- この5つがユーザーとの接点（入口）で起きること。LINEでどこまで寄せられるかを見ていく

表現:
- 5つの体験をカード形式で並べる
- 各カードに一言の価値を添える

### スライド 4: LINE のトーク画面で何が難しいか

| 体験 | LINE のトーク画面での現実 | 難しさ |
|------|--------------------------|--------|
| Streaming | メッセージは完成形で届く。1文字ずつ流れない | 難しい |
| 動的UI生成 | Flex Message は送信時に確定。送った後は変わらない | 難しい |
| 実行前の承認 | Quick Reply / Postback で選択肢は出せる | 工夫すればできる |
| ツール接続 | Bot のサーバー側で外部API を叩ける。MCP Server も作れる | できる |
| 画像理解 | 画像の送受信は標準対応。Vision AI に渡せる | できる |

追加の制約:
- Reply Token は数十秒で失効 → 重い処理は非同期（Push Message）にする必要あり
- 無料プランは月200通の Push Message 制限
- LINE は会話履歴APIを提供しない → 文脈維持はサーバー側で自前構築

話す内容:
- 「難しい」「工夫すればできる」「できる」の3段階
- 全滅ではない。むしろ画像理解とツール接続はそのままいける
- 難しい2つ（Streaming / 動的UI）をどう寄せるかがこの後の本題

表現:
- 赤「難しい」/ 黄「工夫すればできる」/ 緑「できる」で色分け
- 緑が2つあることを目立たせる（LINE にも強みがある）

### スライド 5: 寄せ方の全体像

**トーク画面内の既存APIを役割分担させる。**

| やりたいこと | 使う LINE API |
|-------------|---------------|
| 日常の会話応答 | Reply Message / Push Message |
| リッチな情報表示 | Flex Message（AIに JSON を生成させる） |
| ユーザーの承認・選択 | Quick Reply + Postback |
| 処理中の表示 | Loading Animation |
| 複数の役割の見せ分け | sender（アイコン・名前切替） |
| 画像分析 | 画像受信 → getMessageContent → Vision AI |
| 外部ツール操作 | サーバー側の API 呼び出し / MCP Server |

話す内容:
- 全部トーク画面の中で動く。LIFF（WebView）に飛ばさない前提で考える
- LIFF は最終手段として存在するが、遷移した時点で「LINEのチャット体験」から離れる
- だから今日はトーク画面内でどこまでいけるかにフォーカスする

表現:
- テーブルを翻訳マップとして見せる
- 「トーク画面内で完結」を強調

### スライド 6: 寄せ方① — Streaming の代わりに Loading + 非同期 Push

問題: Messaging API ではトークンが1文字ずつ流れる体験は作れない

寄せ方:
1. メッセージ受信 → **Loading Animation** を表示（最大60秒）
2. サーバー側で AI に問い合わせ（非同期）
3. 結果を **Push Message** で送信
4. 長い回答は **Flex Message** で構造化して見やすく返す

Loading Animation の仕様:
- `showLoadingAnimation` API で「...」アニメーションを表示
- 最大60秒。1対1チャットのみ（グループでは動作しない）

話す内容:
- 1文字ずつ流れる体験は再現できない。これは割り切る
- でも「考え中」が伝わり、結果が構造化されて届く体験は十分作れる
- ChatGPT風のUXを目指すのではなく、「LINE的なAI応答」として成立させる

表現:
- 3ステップのフロー図: Loading → AI処理 → Flex Message で結果表示
- 「ChatGPT風」に寄せるのではなく「LINE風のAI応答」を作る

### スライド 7: 寄せ方② — 動的UIの代わりに Flex Message 動的生成

問題: A2UI / MCP Apps のように、チャット内でUIが動的に変わる体験は作れない

寄せ方:
- AI の Structured Output で **Flex Message の JSON を動的に生成** する
- カード内容はリクエストごとに変わる = 「内容が動的なカード」
- 送信後に変化はしないが、毎回その時点の最適な表示が届く

具体例:
- 検索結果 → カルーセル形式の Flex Message
- 予約確認 → 日時・場所・金額をまとめたカード
- 分析結果 → セクション分けされた要約カード

Flex Message の表現力:
- CSS Flexbox ベースで、ヘッダー・ボディ・フッターの構造
- 画像、ボタン、セパレーター、テキストの組み合わせ
- カルーセルで横スクロールも可能

話す内容:
- Flex Message は静的 JSON だが、AIが毎回生成するなら実質的にレスポンスは動的
- 本物の Generative UI との違い: 送った後に状態が変わるかどうか
- でも多くのユースケースでは「その時点の情報が見やすく届く」で十分

表現:
- AI → Structured Output → Flex Message JSON → LINE カード表示の流れ
- 実際の Flex Message のスクリーンショット

### スライド 8: 寄せ方③ — 実行前承認は Quick Reply + Postback

問題: 最先端では `needsApproval` フラグ一つで承認UIが自動生成される

寄せ方:
- **Flex Message** で実行内容を提示
- **Quick Reply** で「承認」「修正」「キャンセル」を表示
- **Postback** で選択を受け取り、サーバー側で実行

フロー:
> AI: 「明日14時にミーティングを登録します」（Flex Messageで詳細表示）
> Quick Reply: [承認] [時間変更] [キャンセル]
> User: [承認] をタップ
> AI: 「登録しました ✅」

日時が必要な場合:
- Quick Reply の **Datetime Picker アクション** で日時選択UIを呼び出せる

話す内容:
- Human-in-the-Loop の本質は「危ない操作の前に人間を挟む」
- LINE の Quick Reply はタップ1つで承認できる。UIはシンプルだが、承認フローとしては十分
- 実装の注意: Postback data にはセッション ID だけ入れて、実行内容はサーバー側で保持

表現:
- LINE のトーク画面風のモックでフローを見せる
- Quick Reply ボタンが並んでいる画面

### スライド 9: 寄せ方④ — 画像理解はLINEの得意分野 + sender で役割を見せる

**画像理解（Multimodal）:**
- LINE は画像・動画・音声・ファイルの送受信が標準
- `getMessageContent` でバイナリ取得 → Claude Vision / GPT-4V に渡す → Flex で結果を返す
- 「写真を撮って送るだけ」は最高に低い導入障壁
- 例: 料理写真→カロリー推定、名刺→連絡先カード化、レシート→家計簿登録

**sender 切替で役割を見せる:**
- `sender` プロパティでメッセージごとにアイコンと名前を変更可能
- 1つの Bot 内で「検索役」「分析役」「確認役」を見せ分けられる
- ユーザーには「複数のAIが協力してくれてる」ように見える
- ただし A2A（エージェント間プロトコル）とは異なる。内部は1つの Bot

話す内容:
- 画像理解は LINE の強み。最先端と遜色ない体験がトーク内で完結する
- sender 切替はマルチエージェントそのものではないが、役割の可視化として有効
- 誠実に「A2A ではない」と言うと、逆に信頼される

表現:
- 左: 料理写真 → カロリーカードのフロー
- 右: sender 切替で異なるアイコン・名前の吹き出しが並ぶ画面

### スライド 10: まとめ

**LINE のトーク画面で、AIエージェントの入口をどこまで寄せられるか。**

| 最先端の体験 | LINE での寄せ方 | どのくらい寄せられるか |
|-------------|-----------------|----------------------|
| Streaming | Loading Animation + 非同期 Push + Flex | 体験は異なるが実用上は十分 |
| 動的UI生成 | Flex Message を AI が動的生成（Structured Output） | 内容は動的。送信後の変化は不可 |
| 実行前承認 | Quick Reply + Postback | フローとしてはほぼ同等 |
| 画像理解 | 画像受信 + Vision AI | 最先端と遜色なし |
| 役割の見せ分け | sender 切替 | 見た目の分離は可能。A2Aではない |
| ツール接続 | サーバー側実装 / MCP Server | 制約なし |

大事なこと:
- トーク画面内で完結する範囲で寄せる
- 「ChatGPT の再現」を目指すのではなく、「LINE 的なAIエージェントの入口」として設計する
- LINE の強み（導入障壁の低さ、画像送信の自然さ）を活かす
- 寄せきれない体験は、無理にトーク画面に押し込まない

> LINE はAIエージェントそのものにはなれない。
> でも「入口」としてなら、既存APIの組み合わせで思ったより寄せられる。

表現:
- まとめテーブルを大きく表示
- 「寄せられる」を結論として強調
- QRコード: 記事 + ショーケースBot + GitHub
